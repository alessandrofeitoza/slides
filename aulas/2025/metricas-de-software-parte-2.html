Métricas de Software Modernas

Como tipos de task, tamanho, prioridade, tempo e escopo influenciam nas métricas — e critérios de definição de pronto

  • Métricas sugeridas e como medir

Tipos de Task

  • Feature: novas funcionalidades
  • Bug: correções que impactam a execução
  • Adjust: pequenas mudanças visuais ou comportamentais
  • Improvement: melhorias significativas em funcionalidades existentes
  • Quality: qualidade do código / refatoração
  • Discovery: investigação, pouco ou nenhum código
  • Docs: documentação

Como o tipo de task influencia métricas

  • Feature: normalmente aumenta o tamanho médio de PRs, maior esforço de QA e maior lead time
  • Bug: tende a reduzir tempo para entrega (prioridade alta), impacto em MTTR e número de defects escapados
  • Adjust: baixa complexidade, alta throughput quando bem categorizado
  • Improvement: pode aumentar code churn e exigir métricas de qualidade (cobertura, lint)

Como o tipo de task influencia métricas

  • Quality: foca em métricas de manutenção: technical debt, complexidade ciclomática
  • Discovery: medir tempo de investigação e decisões geradas (risks mitigated)
  • Docs: medir cobertura de documentação e tempo de onboarding reduzido

Etapas do desenvolvimento (workflow)

  1. Idea → Product Backlog
  2. Product Backlog → Sprint Backlog
  3. ToDo → InProgress → Review → Test → Done → Closed
  4. Cada transição deve registrar timestamp para calcular cycle time e lead time

Prioridade

  • Very High — prioridade máxima (crítico)
  • High — impacto alto
  • Medium — padrão
  • Low — menor urgência
  • Very Low — backlog de baixo valor

Peso (Weight) — Poker Planning

Sequência Fibonacci utilizada para estimativas

1Tarefa muito simples
2Tarefa simples
3Tarefa mediana
5Tarefa muito complexa

Regra: se uma task receber tamanho 8, deve ser quebrada em duas ou mais tarefas.

Métricas sugeridas

  • Lead time: tempo desde criação até entrega (todas as etapas)
  • Cycle time: tempo desde InProgress até Done
  • Throughput: número de entregas por período
  • PR size: linhas alteradas / arquivos modificados
  • Code churn: proporção de código reescrito
  • Escaped defects: bugs em produção por release
  • MTTR: tempo médio para recuperação de falhas
  • Test coverage & quality gates

Como medir por tipo de task

  • Features: monitorar PR size, review time, QA time e lead time
  • Bugs: priorizar MTTR, taxa de reabertura e tempo até patch
  • Adjust: medir throughput e tempo médio (deve ser baixo)
  • Improvement & Quality: acompanhar code churn, cobertura de testes e technical debt
  • Discovery: medir tempo gasto e número de decisões/trove de valor gerado
  • Docs: tempo para criar/atualizar + impacto em onboarding

Critérios de definição de pronto (Definition of Done)

  • Código implementado e compilando sem erros
  • Testes unitários/integração existentes e passando
  • Revisão de código aprovada (mínimo 1 revisor)
  • Testes manuais em homologação realizados e aprovados
  • Documentação mínima atualizada (se aplicável)
  • Tickets vinculados, changelog e release notes atualizados (se aplicável)
  • Feature flag (quando aplicável) configurada

Exemplos práticos

  • Task: Feature (peso 5) — dividir em sub-tasks de 3 + 2 se exceder PR size
  • Task: Bug (priority Very High) — fluxo: ToDo → InProgress → Review → Done (deploy hotfix)
  • Task: Discovery (peso 1 ou 2) — medir tempo e decisão tomada

Boas práticas

  • Registrar timestamps nas transições do fluxo
  • Criar SLAs internos por tipo/prioridade
  • Monitorar PR size e bloquear merges acima de um limite sem aprovações extras
  • Usar etiquetas para facilmente segmentar métricas (type, priority, weight, area: front/back)

Dashboard sugerido

  • Visão por tipo: throughput, cycle time, média de weight
  • Visão por prioridade: MTTR, defects por release
  • Visão por equipe/área: PR size médio, cobertura de testes
  • Alertas: spikes de lead time, PR size acima do limiar, aumento de escaped defects

Conclusão

  • Categorias claras + regras de estimativa tornam métricas acionáveis
  • Quebre tasks grandes (8+) para previsibilidade
  • Combine métricas de fluxo (lead/cycle) com métricas de qualidade
