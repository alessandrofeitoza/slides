Testes Automatizados

Tão natural quanto a luz do poste 💡

Por que investir em testes?

  • 🐛 Evita bugs em produção
  • 💻 Facilita manutenção e refatoração
  • 📈 Reduz custos a longo prazo
  • 🧠 Documenta o comportamento esperado do sistema
  • 👥 Facilita o trabalho em equipe e onboarding
  • 🔄 Torna refatorações mais seguras
  • 🚨 Detecta falhas rapidamente

Testes Automatizados vs QA

  • Testes Automatizados: escritos por devs, focam no comportamento do código.
  • Testes de QA: realizados por profissionais de qualidade (analistas/testers), muitas vezes de forma manual.

Diferenças principais

  • QA analisa usabilidade, experiência, acessibilidade e casos não previstos
  • Automatizados cobrem o que foi previsto e codificado
  • Ambos são complementares

Teste Caixa Preta vs Caixa Branca

  • Caixa Preta: testa o sistema sem conhecer o código
  • Caixa Branca: testa com conhecimento da lógica interna

Exemplos práticos

  • ✅ Caixa Preta: Verifica se login() retorna erro com senha inválida
  • ✅ Caixa Branca: Verifica se todos os ramos do if dentro de login() são cobertos

Exemplo prático

  • 🧑‍💻 Dev escreve teste automatizado que garante que a função login() funciona
  • 🧪 QA testa login em diferentes navegadores, dispositivos e situações fora do esperado

Tipos de teste

  • Unitário
  • Integração
  • Funcionais
  • E2E

Teste Unitário

  • Testam unidades isoladas do código (funções, métodos ou classes).
  • Não dependem de bancos de dados, APIs ou outras partes externas.
  • Rápidos e fáceis de rodar.
  • Exemplo: Testar se uma função soma(a, b) retorna o valor correto.

Teste de Integração

  • Verificam a interação entre diferentes módulos do sistema.
  • Podem envolver chamadas ao banco de dados, APIs e serviços externos.
  • Mais lentos que os testes unitários.
  • Exemplo: Testar se um método de uma classe consegue buscar e processar dados corretamente de um banco.

Teste Funcional

  • Validam se um fluxo específico do sistema está funcionando conforme esperado.
  • São mais próximos do uso real do software.
  • Exemplo: Testar se um endpoint consegue fazer login com credenciais corretas.

Teste E2E

  • Simulam a experiência real do usuário no sistema completo.
  • Usam ferramentas como Cypress, Selenium ou Playwright.
  • São os mais lentos, pois testam todo o fluxo de ponta a ponta.
  • Exemplo: Testar se um usuário consegue se registrar, fazer login e concluir uma compra.

            npx jest

Testando uma classe real


            class Calculadora {
                soma(a, b) {
                    return a + b;
                }
            }

Testando a Calculadora


            test('Calculadora.soma deve retornar 5 para 2 + 3', () => {
                const calc = new Calculadora();
                expect(calc.soma(2, 3)).toBe(5);
            });

Testes Funcionais

Validação de funcionalidades específicas do sistema

Prós dos Testes Funcionais

  • Cobrem cenários reais de uso
  • Ajudam a detectar bugs em fluxos importantes
  • Podem ser automatizados e integrados no CI

Contras dos Testes Funcionais

  • Mais lentos que testes unitários
  • Podem ser instáveis se o ambiente mudar
  • Difíceis de manter em sistemas grandes

Testes End-to-End (E2E)

Simulam o comportamento do usuário do início ao fim

Ferramentas comuns

  • 🔧 Cypress (JavaScript)
  • 🔧 Playwright
  • 🔧 Selenium

Prós dos Testes E2E

  • Cobre todo o sistema: frontend + backend
  • Simula o comportamento do usuário final
  • Evita regressões em funcionalidades críticas

Contras dos Testes E2E

  • 👎 Lentos (envolvem múltiplas camadas)
  • ⚠️ Suscetíveis a falhas por instabilidade do ambiente
  • 💸 Requerem mais infraestrutura (ex: servidores, CI robusto)
  • 📉 Mais difíceis de depurar quando falham

Boas práticas

  • Testes devem ser independentes
  • Nome dos testes deve ser claro
  • Utilize setUp() para preparar o ambiente
