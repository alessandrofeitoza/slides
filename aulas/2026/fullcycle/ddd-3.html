Parte 3

Entities, Value Objects e Repositories

Modelando comportamento e persistência

Entities

Objetos com identidade

Nem todo objeto é igual

Alguns objetos importam pelo que são, não apenas pelos dados que possuem.

Identidade importa.

O que é uma Entity?

Um objeto identificado por uma identidade única, cuja continuidade importa ao longo do tempo.

  • Possui ID
  • Pode mudar de estado
  • Representa algo rastreável

Exemplo

Usuário

Mesmo mudando nome ou email, continua sendo o mesmo usuário.

Pedido

Status muda, mas o pedido permanece.

Entity em código


class Pedido
{
    private string $id;
    private string $status;
}

O ID define identidade.

Value Objects

Objetos definidos por valor

Nem tudo precisa de ID

Alguns objetos importam apenas pelo valor que carregam.

O que é um Value Object?

Objeto imutável definido pelos seus atributos.

  • Sem identidade
  • Imutável
  • Comparado por valor

Exemplos

CEP
Dinheiro
CPF
Endereço

Entity vs Value Object

Entity Value Object
ID Sim Não
Mutável Sim Preferencialmente não
Comparação Identidade Valor

Exemplo de Value Object


class Dinheiro
{
    public function __construct(
        private int $valorEmCentavos
    ) {}
}

Dois objetos Dinheiro com mesmo valor representam a mesma coisa.

Repositories

Persistência sem poluir domínio

O problema


Pedido::where(...)
DB::table(...)
mysql_query(...)

Domínio acoplado ao banco.

Repository Pattern

Repository representa uma coleção de objetos do domínio.

O domínio não sabe onde os dados vivem.

Analogia

Biblioteca

Você pede um livro. Não precisa saber onde está.

Repository

Você pede um Pedido. Não importa se vem de SQL, API ou cache.

Interface de Repository


interface PedidoRepository
{
    public function buscarPorId(
        string $id
    ): Pedido;

    public function salvar(
        Pedido $pedido
    ): void;
}

Resumo da Parte 3

  • Entities possuem identidade
  • Value Objects representam valor
  • Repositories isolam persistência
  • Domínio permanece limpo
