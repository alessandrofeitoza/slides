O ambiente é o grande vilão

O Cenário do Desastre

  • .env versionado.
  • Chaves "Hardcoded" no código.
  • Ambiente local diferente da produção.
  • Logs expondo dados de clientes.

Código vs Configuração

Código

Lógica de negócio, algoritmos, rotas. É igual em qualquer lugar.

Config

Portas, URLs de banco, níveis de log. Muda conforme o lugar.

E os Segredos?

Configurações que, se vazadas, comprometem o negócio.

  • Tokens de API (Stripe, AWS, Sendgrid).
  • Senhas de Bancos de Dados.
  • Certificados SSL / Chaves Privadas.

1. Desenvolvimento (DEV)

  • Onde o código nasce.
  • Foco: Velocidade e Debug.
  • Config: DB_HOST=localhost, DEBUG=true.

2. Staging (Homologação)

  • O espelho da realidade.
  • Foco: Testar o deploy antes do cliente ver.
  • Config: Banco de dados isolado, mas com dados reais (anônimos).

3. Produção (PROD)

  • Onde o dinheiro entra.
  • Foco: Segurança, Estabilidade e Performance.
  • Config: DEBUG=false, Logs sanitizados.

O Padrão: .env

O arquivo que você nunca deve compartilhar.


        PORT=3000
        DATABASE_URL=postgres://user:pass@localhost:5432/db
        STRIPE_KEY=sk_test_aaaaa

O Vigia


        # .gitignore
        .env
        node_modules/
        dist/
        .DS_Store

Sem isso, adianta bulhufas nenhuma.

O Mapa do Tesouro

Documente o que o projeto precisa para rodar.


        # .env.example
        PORT=3000
        DATABASE_URL=
        STRIPE_KEY=

Valores vazios ou genéricos apenas!

Implementação: Node.js


        require('dotenv').config();

        const port = process.env.PORT;
        console.log(`Rodando na porta ${port}`);

Implementação: PHP


        $dotenv = Dotenv\Dotenv::createImmutable(__DIR__);
        $dotenv->load();

        $dbHost = $_ENV['DB_HOST'];

Troca de Ambiente

O que acontece quando você muda de Dev para Prod?

  • Variáveis faltantes derrubam o app.
  • Permissões de escrita em pastas.
  • Conexões de rede (Firewalls).

Aula 20: Segurança de Ambiente

Hardening e Blindagem de Aplicações

Defesa em Profundidade

  1. Segurança no Código (Sanitização).
  2. Segurança no Ambiente (Configs).
  3. Segurança na Infra (Rede/Firewall).

APP_DEBUG = TRUE

Fatal Error: Cannot connect to DB.
at /var/www/html/config/database.php:45
'password' => '123456'
'host' => '10.0.0.5'

Isso é um convite para hackers.

Vazamento em Logs


        // Ruim
        console.log("Usuário tentando login:", user);
        // O log salva a senha em texto plano no disco!

Sempre sanitize os logs antes de salvar.

O Cavalo de Troia

Você usa bibliotecas vulneráveis?


        # Verifique agora:
        npm audit
        # ou
        composer audit

GitHub Secrets

Onde colocar as chaves se o .env não sobe?

  • Settings > Secrets and Variables > Actions.
  • Injetadas durante o Deploy via Pipeline.

Segurança em Containers

Nunca rode como Root!


        # Dockerfile Profissional
        FROM node:alpine
        RUN adduser -D appuser
        USER appuser
        COPY . .
        CMD ["node", "server.js"]

Sistema de Arquivos

Em produção, seu container deve ser "Imutável".


        # docker-compose.yml
        services:
          app:
            read_only: true
            tmpfs:
              - /tmp

🔍 Desafio de Auditoria

  1. Rode npm audit ou composer audit.
  2. Quantas vulnerabilidades existem?
  3. O que a biblioteca vulnerável faz?

Menor Privilégio

Dê apenas o acesso necessário.

  • Usuário do Banco só pode dar SELECT/INSERT?
  • A aplicação pode apagar tabelas? (Espero que não).

Varredura de Segredos

Ferramentas que impedem o git push se houver uma chave no código:

  • gitleaks
  • trufflehog
  • GitHub Secret Scanning

Organização Final do Projeto

  • 📁 src/ (Código lógico)
  • 📄 .env.example (Documentação de chaves)
  • 📄 .gitignore (Proibindo o .env)
  • 📄 Dockerfile (Sem usuário root)
  • 📄 README.md (Instruções de setup)

O Checklist de Ouro

  • [ ] .env ignorado no Git.
  • [ ] APP_DEBUG=false em produção.
  • [ ] Logs não registram senhas.
  • [ ] Dependências auditadas e sem High Vulnerabilities.
  • [ ] Container sem privilégios de root.
  • [ ] Chaves de API no GitHub Secrets (não no código).

Dever de Casa

Refatore seu projeto atual aplicando o checklist anterior. Envie o link do repositório com o arquivo README.md atualizado com instruções de ambiente.
