Aula 17: Cultura DevOps

O Projeto Fênix

Quebrando Silos e Criando Fluxo

Inicie perguntando quem já sentiu que o trabalho é só apagar incêndio.

O Cenário: Parts Unlimited

  • Empresa à beira da falência.
  • Projeto Fênix: A última esperança.
  • Realidade: Atrasado, instável e caro.
"TI não é um departamento; é a competência estratégica da empresa."

O "Muro da Confusão"

DEV

Meta: Mudança
(Novas Features)

🧱

OPS

Meta: Estabilidade
(Manter no Ar)

Resultado: Culpa, lentidão e silos.

Os 4 Tipos de Trabalho

  1. Projetos de Negócio: O que gera receita.
  2. Projetos de TI: Infra, updates, segurança.
  3. Mudanças (Changes): Tickets, pequenos pedidos.
  4. Trabalho Não Planejado: O vilão (Bugs, quedas).

Quanto mais trabalho não planejado, menos tempo para o negócio.

O 1º Caminho: FLUXO

Da Esquerda para a Direita (Dev → Ops)

Como acelerar?

  • Tornar o trabalho visível (Kanban).
  • Reduzir o tamanho dos lotes.
  • Limpar o "WIP" (Work In Progress).

O 2º Caminho: FEEDBACK

Da Direita para a Esquerda (Ops → Dev)

  • O sistema deve avisar quando falha.
  • Corda de Andon: Qualquer um pode parar a linha.
  • Corrigir na fonte, não "deixar para depois".

O 3º Caminho: APRENDIZADO

Cultura de Experimentação e Risco

  • Post-mortems sem culpa (Blameless).
  • Transformar erros em conhecimento.
  • Injetar falhas para treinar a resiliência.

O "Gargalo" chamado Brent

Brent é o cara que sabe tudo e resolve tudo.

Problema:

Nada anda sem o Brent. Ele é o gargalo do sistema.

Se a informação não está no processo, ela não existe.

DevOps é sobre PESSOAS

Ferramentas (Docker, IA, etc) servem à Cultura do bom desenvolvimento.

Parte 2: Do Fluxo à Segurança

Aula 19: Ambientes, Configurações e Segredos

O Problema: "Na minha máquina funciona"

  • Diferenças entre Dev, Staging e Produção.
  • Configurações manuais = Deriva de Configuração.
  • O risco de "brincar" de Deus nos servidores.

1. Imutabilidade da Infraestrutura

Em vez de consertar um servidor, nós o substituímos.

"Servidores devem ser tratados como Gado, não como Pets."

  • Pets: Têm nomes, cuidados especiais e dor quando morrem.
  • Gado: São idênticos, numerados e substituíveis.

2. Gestão de Configuração

Onde guardamos as variáveis de ambiente?

Método Risco Recomendação
Hardcoded (Código) Altíssimo Nunca fazer.
Arquivos .env locais Médio Apenas para Dev.
Gerenciadores de Segredos Baixo Padrão de Indústria.

3. O Perigo do "Leaking"

O que nunca deve ir para o Git?

API Keys
Senhas de Banco
Certificados SSL
Tokens de Acesso

A internet é varrida por bots em segundos à procura de segredos expostos.

4. Princípio do Menor Privilégio

A segurança começa no acesso.

  • A aplicação só acessa o que precisa.
  • O desenvolvedor não precisa de root em produção.
  • RBAC: Role-Based Access Control.

5. Cofres Digitais (Secret Management)

Ferramentas para centralizar a segurança:

HashiCorp Vault
O padrão ouro.
AWS Secrets Manager
Cloud Native.
GitHub Secrets
Para CI/CD.

6. DevSecOps: Shift Left

Trazer a segurança para o início do fluxo.

Segurança não é um "checklist final", é parte do código.

7. Ambientes Efêmeros

O poder dos containers (Docker/K8s).

  • Ambientes que nascem para um teste e morrem em seguida.
  • Garantia de que o ambiente de teste é idêntico ao de produção.

8. Auditoria e Rastreabilidade

"Quem alterou a variável de banco às 03:00 da manhã de domingo?"

Logs não são apenas para erros, são para conformidade.

Conclusão: O Próximo Passo

Ambientes seguros permitem velocidade com confiança.
