Maturidade de Software

Por que algumas equipes entregam valor continuamente e outras vivem apagando incêndios?

Um problema antigo

Nas décadas de 1960 e 1970, projetos de software começaram a ficar maiores, mais caros e mais complexos.

  • atrasos constantes
  • orçamentos estourados
  • sistemas instáveis
  • alta dependência de indivíduos
  • baixa previsibilidade

A Crise do Software

1968

Conferência da OTAN populariza o termo "Software Crisis"

1970

Engenharia de Software começa a se consolidar como disciplina

1980+

Organizações buscam modelos para medir qualidade e previsibilidade

O que significa maturidade?

Maturidade não mede apenas tecnologia.

Mede a capacidade de uma organização repetir resultados com qualidade.

Capability Maturity Model (CMM)

Criado pelo Software Engineering Institute (SEI), da Carnegie Mellon University.

  • final da década de 1980
  • encomendado pelo Departamento de Defesa dos EUA
  • objetivo: avaliar fornecedores de software

Os 5 níveis do CMM

1

2

3

4

Gerenciado

Nível 1 - Inicial

"Cada projeto é uma aventura."
  • processos inexistentes
  • dependência de pessoas específicas
  • resultados imprevisíveis

Nível 5 - Otimizado

"A organização aprende continuamente."
  • métricas
  • melhoria contínua
  • automação
  • evolução baseada em dados

Nasce o CMMI

Em 2002 surge o CMMI (Capability Maturity Model Integration).

  • integra diversos modelos existentes
  • mais flexível
  • mais adequado para organizações complexas

CMM x CMMI

E o Brasil?

Pequenas e médias empresas tinham dificuldade para implementar modelos como CMMI.

MPS.BR

Melhoria de Processo do Software Brasileiro

  • lançado em 2003
  • coordenado pela SOFTEX
  • adaptado à realidade brasileira
  • menor custo de implementação

Overengineering

Criar uma solução muito mais complexa do que o problema exige.

  • aumenta custos
  • aumenta manutenção
  • reduz velocidade de entrega
  • gera dívidas técnicas prematuras

Exemplo real


class OrderController
{
    public function store(Request $request)
    {
        $order = Order::create($request->all());

        Mail::send(...);

        PaymentGateway::charge(...);

        return response()->json($order);
    }
}

Sim.

O sistema continua funcionando...

O que é Débito Técnico?

Uma decisão que acelera o presente, mas aumenta o custo do futuro.

Exemplo simples


if ($user->type == 1) {
    //
}

Nada assustador até aqui.

Alguns meses depois...


if ($user->type == 1) {
    //
}
elseif ($user->type == 2) {
    //
}
elseif ($user->type == 3) {
    //
}
elseif ($user->type == 4) {
    //
}

Dois anos depois...


if (...) {
    ...
}
elseif (...) {
    ...
}
elseif (...) {
    ...
}
elseif (...) {
    ...
}
elseif (...) {
    ...
}
elseif (...) {
    ...
}

Todos conhecem esse código.

Mas o sistema funciona!

Sim.

Então por que isso é um problema?

O custo escondido

  • mais difícil de entender
  • mais difícil de testar
  • mais difícil de modificar
  • mais chance de quebrar regras existentes
  • mais tempo para onboarding

Juros do débito técnico

Hoje

  • 5 minutos
  • 1 ajuste rápido

Daqui a 2 anos

  • 2 dias
  • vários impactos
  • risco elevado

Nem todo débito técnico é ruim

Muitas startups sobrevivem justamente porque assumem débitos técnicos conscientemente.

O problema é quando a dívida deixa de ser controlada.

Exemplo clássico

MVP

  • entrega rápida
  • validação de mercado
  • baixo investimento

Escala

  • crescimento de usuários
  • novos requisitos
  • necessidade de evolução

Débito Técnico x Falta de Qualidade

Débito Técnico Falta de Qualidade
decisão consciente erro ou negligência
possui justificativa não possui justificativa
possui plano futuro não possui plano

Como identificar débitos técnicos?

  • duplicação de código
  • classes gigantes
  • acoplamento excessivo
  • baixa cobertura de testes
  • dependências obsoletas
  • processos manuais recorrentes

Uma alternativa melhor


interface PaymentMethod
{
    public function pay(): void;
}

O objetivo não é apenas deixar o código bonito.

O verdadeiro objetivo

Tornar mudanças futuras mais baratas.

Maturidade Arquitetural e Organizacional

O maior desafio não é técnico

A maioria dos problemas de software nasce da interação entre pessoas, processos e tecnologia.

Uma reflexão

O que acontece quando apenas uma pessoa conhece uma parte crítica do sistema?

O Fator Ônibus

Quantas pessoas poderiam sair da empresa antes que o projeto parasse?

Quanto menor o número, maior o risco.

Nível 1 - Heroísmo

"Deixa que eu resolvo."

  • conhecimento concentrado
  • dependência de indivíduos
  • decisões informais
  • incêndios constantes

Nível 2 - Processos

"Vamos repetir o que funciona."

  • padronização
  • documentação mínima
  • fluxos definidos
  • menos dependência individual

Nível 3 - Automação

"Se é repetitivo, pode ser automatizado."

  • pipelines CI/CD
  • testes automatizados
  • quality gates
  • infraestrutura como código

Nível 4 - Observabilidade

"Entender o sistema em produção."

  • logs estruturados
  • métricas
  • tracing distribuído
  • alertas automatizados

Nível 5 - Evolução Contínua

"Mudanças sem trauma."

  • aprendizado contínuo
  • feedback rápido
  • decisões baseadas em dados
  • adaptação constante
