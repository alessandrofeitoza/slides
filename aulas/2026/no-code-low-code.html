Code, No-Code, e Low-Code

Do problema real ao código

Abordagens diferentes para problemas diferentes

⚡ No Code

  • Construção 100% visual
  • Não exige conhecimento técnico profundo
  • Extremamente rápido para validar ideias
  • Limitações fortes de escala e regra de negócio

Ideal para: protótipos, landing pages, validação de negócio

  • WIX
  • FlutterFlow
  • AirTable

🧩 Low Code

  • Desenvolvimento visual + código quando necessário
  • Acelera entregas sem perder totalmente o controle
  • Dependência da plataforma
  • Customização limitada fora do "caminho feliz"

Ideal para: MVPs, sistemas internos, automações mais robustas

  • Zeev
  • SalesForce
  • AppBuilder

🧠 Programação Convencional

  • Código escrito manualmente (PHP, JS, Java, etc.)
  • Controle total da regra de negócio
  • Alta flexibilidade e escalabilidade
  • Maior custo inicial (tempo + pessoas)

Ideal para: sistemas complexos, regras específicas, longo prazo

📊 Comparação Direta

Critério Convencional Low Code No Code
Flexibilidade Alta Média Baixa
Velocidade Inicial Baixa Alta Muito Alta
Escalabilidade Alta Média Baixa
Dependência de Ferramenta Nenhuma Alta Total

💡 A pergunta certa não é “qual é melhor?”

  • Qual é o problema?
  • Qual o prazo?
  • Quem vai manter isso?
  • Qual o custo do erro?

Onde entram HTML e CSS?

  • HTML define a estrutura da interface
  • CSS define aparência e comportamento visual
  • No Code e Low Code geram HTML + CSS por baixo
  • Programação convencional escreve isso de forma explícita

HTML — Estrutura

  • Define o esqueleto da página
  • Organiza conteúdo: texto, tabelas, formulários
  • Sem CSS é funcional, mas feio
  • Base de qualquer aplicação web

HTML responde: o que existe na tela?

CSS — Aparência

  • Define cores, espaçamentos e fontes
  • Controla layout (grid, flex, responsividade)
  • Transforma HTML cru em interface agradável
  • Sem HTML, CSS não existe

CSS responde: como isso aparece?

HTML + CSS no No Code e Low Code

  • Ferramentas visuais geram HTML automaticamente
  • O CSS vem pronto, mas limitado
  • Quanto menos código, menos controle
  • Saber HTML/CSS evita dependência total da ferramenta

COMPONENTES

O que é um componente de interface?

  • Bloco reutilizável de UI
  • Combina HTML + CSS (+ JS)
  • Resolve um problema específico de interação
  • Presente em código, Low Code e No Code

Componente: Tabela

  • Exibe dados estruturados
  • Muito usada em sistemas administrativos
  • Pode ter paginação, filtros e ordenação
  • Complexidade cresce rápido

Quando usar: dados comparáveis e listagens

Componente: Card

  • Agrupa informações relacionadas
  • Visualmente simples e reutilizável
  • Muito comum em dashboards
  • Facilita leitura e escaneabilidade

Quando usar: resumos, KPIs, blocos de informação

Componente: Dropdown

  • Economiza espaço na interface
  • Usado para seleção ou navegação
  • Depende de JavaScript
  • Pode causar problemas de usabilidade

Quando usar: muitas opções, pouco espaço

Componente: Modal

  • Interrompe o fluxo do usuário
  • Usado para confirmação ou ações críticas
  • Precisa ser usado com cuidado
  • Muito comum em CRUDs

Regra de ouro: só use quando necessário

Componente: Menu Lateral / Drawer

  • Principal estrutura de navegação do sistema
  • Pode ser fixo ou retrátil
  • Muito usado em dashboards e sistemas internos
  • Impacta diretamente a usabilidade

Cuidado: excesso de opções confunde

Componente: Input

  • Permite entrada de dados pelo usuário
  • Pode ter validação, máscara e feedback visual
  • Base de qualquer formulário
  • Simples por fora, crítico por dentro

Regra: erro precisa ser claro

Componente: Select

  • Usado para escolher uma opção entre várias
  • Pode ser nativo ou customizado
  • Customização excessiva pode piorar UX
  • Muito comum em filtros e formulários

Dica: poucas opções? prefira radio

Componente: Tabs

  • Organiza conteúdos relacionados
  • Evita múltiplas telas desnecessárias
  • Mantém contexto do usuário
  • Muito usada em telas de detalhe

Use quando: o conteúdo é do mesmo domínio

Componente: Botão

  • Dispara ações
  • Pode ser primário, secundário ou destrutivo
  • Cor comunica intenção
  • Elemento mais clicado da interface

Erro comum: muitos botões primários

Checkbox & Radio

  • Checkbox: múltiplas escolhas
  • Radio: escolha única
  • Simples, mas essenciais
  • Muito usados em formulários e filtros

UX: escolha errada confunde o usuário

Componente: Alert / Feedback

  • Informa sucesso, erro ou aviso
  • Guia o usuário após uma ação
  • Cores têm significado semântico
  • Essencial para experiência

Sem feedback: o usuário se perde

Componentes não são só visuais

  • Carregam regras de interação
  • Impactam diretamente a experiência
  • Precisam ser consistentes
  • São reutilizáveis por definição

O que muda entre as abordagens?

  • Código: você cria e controla o componente
  • Low Code: você ajusta o componente
  • No Code: você aceita o componente
  • Menos código = menos decisão

Entender componentes é entender produto

Amarrando tudo

  • HTML estrutura
  • CSS apresenta
  • Componentes organizam
  • Ferramentas só aceleram decisões

Quem entende a base, escolhe melhor a ferramenta
