Aula 02: Automação com Cypress

Foco em Resiliência e Continuous Testing

O que é Teste E2E?

  • Validação do fluxo completo do usuário.
  • Simula o comportamento humano no navegador.
  • Testa a integração de todos os sistemas (Frontend, Backend, DB, APIs).

Cypress: A Nova Era

  • Roda dentro do navegador (não usa Selenium/WebDriver).
  • Acesso direto à rede, DOM e Storage.
  • Time Travel: Debug visual passo a passo.
  • Automatic Waiting (adeus `sleep(5000)`).

O que conseguimos testar?

  • Login
  • Cadastro
  • CRUD
  • Fluxos completos
  • Requisições HTTP
  • Cookies
  • LocalStorage
  • Responsividade

Instalação


npm install cypress --save-dev
  

npx cypress open

O Cypress irá criar automaticamente a estrutura inicial do projeto.

Estrutura Básica


cypress/
 ├── e2e/
 ├── fixtures/
 ├── support/
 └── downloads/
  • e2e → testes
  • fixtures → dados mockados
  • support → comandos auxiliares

describe('Login', () => {

  it('deve realizar login', () => {

    cy.visit('/login')

    cy.get('[data-cy=email]')
      .type('admin@email.com')

    cy.get('[data-cy=senha]')
      .type('123456')

    cy.get('[data-cy=submit]')
      .click()

    cy.contains('Dashboard')
  })

})

10 Boas Práticas (Cypress)

1. Use Seletores Dedicados

Não use classes CSS ou IDs que mudam com o design. Use atributos data-*.


// ❌ RUIM (Fragil)
cy.get('.btn-submit-login-v2').click()

1. Use Seletores Dedicados

Não use classes CSS ou IDs que mudam com o design. Use atributos data-*.


// ❌ RUIM (Fragil)
cy.get('.btn-submit-login-v2').click()

// ✅ BOM (Resiliente)
cy.get('[data-cy="submit-button"]').click()

2. Login via API (Programático)

Não use a UI para logar antes de cada teste. Use cy.request() para ganhar tempo.


beforeEach(() => {
  cy.request('POST', '/api/login', { user: 'admin', pass: '123' })
    .then((res) => {
      window.localStorage.setItem('token', res.body.token)
    })
})

3. Testes Atômicos e Isolados

Um teste não deve depender do resultado do teste anterior.

Regra: Cada teste deve ser capaz de rodar sozinho.

it('deve criar um post', () => { ... })
it('deve editar um post', () => {
    // Em vez de depender do teste acima,
    // crie o post via API no beforeEach
})

4. Jamais use cy.wait(Número)

Esperar 5 segundos é perda de tempo no CI/CD. Espere por elementos ou rotas.


// ❌ RUIM
cy.wait(5000)

// ✅ BOM (Espera o elemento existir)
cy.get('.loading-spinner').should('not.exist')

5. Use cy.intercept para Mockar

Controle o backend para testar cenários de erro (Edge Cases).


cy.intercept('GET', '/users', { statusCode: 500 }).as('getUsers')
cy.visit('/dashboard')
cy.wait('@getUsers')
cy.contains('Erro ao carregar dados').should('be.visible')

6. Abstraia com Custom Commands

Crie comandos para ações repetitivas, mas não exagere.


// em cypress/support/commands.js
Cypress.Commands.add('getByData', (selector) => {
  return cy.get(`[data-cy=${selector}]`)
})

// No teste
cy.getByData('username-input').type('admin')

7. Configure a baseUrl

Nunca use URLs completas nos testes. Facilite a troca de ambiente (Dev/Staging/Prod).


// cypress.config.js
baseUrl: 'http://localhost:3000'

// No teste
cy.visit('/login') // Navega para localhost:3000/login

8. Use Aliases (@)

Facilite a leitura e o reuso de elementos ou requisições.


cy.get('[data-cy="user-row"]').first().as('firstUser')

// ... outras ações ...

cy.get('@firstUser').click()

9. Assertions de Estado

Não teste apenas se o botão existe, mas se ele está habilitado/visível.


cy.get('[data-cy="submit"]')
  .should('be.visible')
  .and('not.be.disabled')
  .and('have.text', 'Salvar')

10. Artifacts para DevOps

No pipeline, habilite gravação de vídeo e fotos em caso de falha.


// cypress.config.js
module.exports = {
  video: true,
  screenshotOnRunFailure: true,
  trashAssetsBeforeRuns: true
}

Desafio Prático

1. Instalar Cypress: npm install cypress

2. Criar um fluxo de login usando as boas práticas.

3. Rodar em modo Headless: npx cypress run

O Maior Problema do E2E

Flaky Tests

Testes que falham sem motivo aparente.

  • Timeouts
  • Dependência de ambiente
  • Selectors frágeis
  • Race conditions

Boas Práticas com Cypress #01

Evite seletores frágeis

Ruim


cy.get('.btn.btn-primary')
cy.get('#login > div:nth-child(2)')

Bom


cy.get('[data-cy=submit]')
cy.get('[data-cy=email]')

Boas Práticas com Cypress #02

Não use waits fixos

Ruim


cy.wait(5000)

Bom


cy.intercept('GET', '/users')
  .as('getUsers')

cy.wait('@getUsers')

Boas Práticas com Cypress #03

Teste comportamento, não implementação

Ruim


cy.window()
  .its('store.state.user')

Bom


cy.contains('Olá Alessandro')
cy.contains('Dashboard')

Boas Práticas com Cypress #04

Crie comandos customizados


// commands.js

Cypress.Commands.add(
  'login',
  (email, senha) => {

    cy.visit('/login')

    cy.get('[data-cy=email]')
      .type(email)

    cy.get('[data-cy=senha]')
      .type(senha)

    cy.get('[data-cy=submit]')
      .click()
  }
)
  

cy.login(
  'admin@email.com',
  '123456'
)

Boas Práticas com Cypress #05

Evite dependência entre testes

Ruim


it('cria usuário')

it('edita usuário criado')

Bom


beforeEach(() => {
  cy.criarUsuario()
})

Boas Práticas com Cypress #06

Use fixtures para dados


// fixtures/usuario.json

{
  "email": "teste@email.com",
  "senha": "123456"
}
  

cy.fixture('usuario')
  .then((usuario) => {

    cy.login(
      usuario.email,
      usuario.senha
    )
  })

Boas Práticas com Cypress #07

Faça assertions explícitas

Fraco


cy.get('.message')

Melhor


cy.get('[data-cy=message]')
  .should(
    'contain',
    'Usuário criado'
  )

Boas Práticas com Cypress #08

Isole chamadas externas


cy.intercept(
  'GET',
  'https://api.externa.com/users',
  {
    statusCode: 200,
    body: [
      { nome: 'Alessandro' }
    ]
  }
)

Evita testes instáveis por causa de APIs de terceiros.

Boas Práticas com Cypress #09

Page Object básico


// login.page.js

export class LoginPage {

  visitar() {
    cy.visit('/login')
  }

  preencherEmail(email) {
    cy.get('[data-cy=email]')
      .type(email)
  }

  preencherSenha(senha) {
    cy.get('[data-cy=senha]')
      .type(senha)
  }

  enviar() {
    cy.get('[data-cy=submit]')
      .click()
  }
}

Boas Práticas com Cypress #10

Evite testes gigantes

Ruim


it('fluxo completo do sistema')

Bom

it('logout')
it('editar perfil')

Testes menores falham menos, executam mais rápido e facilitam debug.

Aula 01: Fundamentos e Mindset DevOps

Por que investir em testes?

  • 🐛 Evita bugs em produção
  • 💻 Facilita manutenção e refatoração
  • 📈 Reduz custos a longo prazo
  • 🧠 Documenta o comportamento esperado do sistema
  • 👥 Facilita o trabalho em equipe e onboarding
  • 🔄 Torna refatorações mais seguras
  • 🚨 Detecta falhas rapidamente

O Cenário DevOps

  • Fim dos silos (Dev | QA | Ops)
  • Feedback contínuo
  • Time to Market vs Qualidade

Por que testar cedo? (Shift-Left)

Quanto mais tarde um bug é encontrado, mais caro ele custa.

"Encontrar um erro em produção pode ser 100x mais caro do que na fase de design." - Lei de Boehm

Estratégias

Pirâmide

  • Mais testes unitários
  • Poucos E2E
  • Execução rápida
  • Baixo custo

Troféu

  • Foco em integração
  • Menos mocks desnecessários

Shift Left & Shift Right

Shift Left

Encontrar problemas o mais cedo possível.

Shift Right

Continuar validando comportamento em produção.

Observabilidade, feature flags, chaos engineering, monitoramento e feedback contínuo.

A Pirâmide de Testes

E2E (Topo): Poucos e lentos

Integração (Meio): Moderados

Unitários (Base): Muitos e rápidos

Objetivo: ROI máximo e feedback rápido.

Qualidade do Teste: Princípios FIRST

  • Fast (Rápido)
  • Independent (Isolado)
  • Repeatable (Repetível)
  • Self-validating (Auto-validável)
  • Timely (No tempo certo)

TDD: Test-Driven Development

  1. RED: Escreva um teste que falha
  2. GREEN: Escreva o código para passar
  3. REFACTOR: Melhore o código
