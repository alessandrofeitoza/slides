O que é um Teste Unitário?

Um teste unitário valida uma pequena unidade do sistema de forma isolada.

  • Uma função
  • Um método
  • Uma regra de negócio
  • Uma classe

Objetivo Principal

Garantir que a regra de negócio funcione corretamente sem depender do resto do sistema.

Não depende de

  • Banco
  • Internet
  • API externa
  • Frontend
  • Docker

Normalmente testa

  • Cálculos
  • Validações
  • Transformações
  • Fluxos internos
  • Regras de negócio

Exemplo sem teste


function calcularDesconto(valor, percentual) {
    if (percentual > 50) {
        throw new Error('Desconto inválido');
    }

    return valor - (valor * percentual / 100);
}

Como garantir que essa regra nunca vai quebrar?

Agora com Teste Unitário


describe('calcularDesconto', () => {
  it('deve aplicar desconto corretamente', () => {
     const resultado = calcularDesconto(100, 10);

     expect(resultado).toBe(90);
  });
});

Testando Cenários de Erro


describe('calcularDesconto', () => {
    it('deve lançar erro para desconto inválido', () => {
        expect(() => {
            calcularDesconto(100, 80);
        }).toThrow('Desconto inválido');
    });
});

Bons testes validam:

  • Cenário feliz
  • Cenário de erro
  • Casos extremos

O teste não garante não ter bugs

Garante que eles foram encontrados

Por que Testes Unitários são importantes?

  • Detectam bugs rapidamente
  • Ajudam no refactor
  • Dão confiança para alterar código
  • Melhoram arquitetura
  • Servem como documentação viva
Código sem teste normalmente fica “frágil”.

Problema comum

Funções gigantes


async function finalizarCompra() {

    // consulta banco
    // chama gateway
    // envia email
    // atualiza estoque
    // gera nota fiscal
    // envia webhook
}

Quanto mais acoplado o código, mais difícil testar.

Código mais testável


function calcularFrete(peso) {

    if (peso <= 1) {
        return 10;
    }

    return 20;
}

Funções pequenas e previsíveis são:

  • Mais fáceis de testar
  • Mais fáceis de manter
  • Mais fáceis de entender

Unitário vs API vs E2E

Unitário

  • Mais rápido
  • Mais isolado
  • Mais barato
  • Foco em regra

API / E2E

  • Mais próximo da realidade
  • Mais lento
  • Mais caro
  • Foco em integração

Resumo

  • Teste unitário valida pequenas partes do sistema
  • Deve ser rápido e isolado
  • Ajuda a evoluir o sistema com segurança
  • É a base da qualidade de software
Testes não existem para provar que o sistema funciona.
Eles existem para detectar quando ele quebra.

Testes de Integração

Validam se diferentes partes do sistema funcionam corretamente juntas.

Aqui começamos a sair do isolamento puro.

O que um teste de integração valida?

  • Controller + Service
  • Service + Banco
  • API + ORM
  • Fila + Worker
  • Microserviço + Mensageria
O foco deixa de ser “uma função” e passa a ser “a comunicação entre módulos”.

Fluxo comum

Request HTTP

→

Controller

→

Service

→

Banco/API

O teste verifica se essas peças se conversam corretamente.

Diferença Principal

Teste Unitário

  • Isolado
  • Sem banco
  • Sem API
  • Muito rápido

Teste de Integração

  • Integra componentes
  • Pode acessar banco
  • Pode subir servidor
  • Mais próximo da realidade

Exemplo de Cenário

Um usuário faz cadastro:

  1. Request chega na API
  2. Controller recebe dados
  3. Service valida
  4. ORM salva no banco
  5. API retorna resposta

O teste de integração valida esse fluxo.

Exemplo com Supertest


const request = require('supertest');
const app = require('./app');

describe('POST /usuarios', () => {

    it('deve criar usuário', async () => {

        const response = await request(app)
            .post('/usuarios')
            .send({
                nome: 'Alessandro',
                email: 'teste@email.com'
            });

        expect(response.statusCode)
            .toBe(201);

    });

});

O que está sendo testado aqui?

  • Rota HTTP
  • Controller
  • Validações
  • Integração com service
  • Resposta da API
Já é muito mais próximo do comportamento real do sistema.

Banco real ou fake?

Banco Fake

  • Mais rápido
  • Mais simples
  • Menos realista

Banco Real

  • Mais confiável
  • Mais lento
  • Mais próximo da produção

Muitos times usam SQLite em memória nos testes.

Problema comum


it('deve cadastrar usuário', async () => {

    // depende da internet
    // depende do banco
    // depende da API externa
    // depende do Redis
    // depende do RabbitMQ

});

Testes lentos e frágeis quebram pipelines.

  • Timeouts

Boas práticas

  • Testes devem ser independentes
  • Evite depender de internet
  • Use massa de dados previsível
  • Limpe banco após cada teste
  • Não teste tudo via E2E
  • Integração ≠ produção completa

Resumo

  • Testes de integração validam comunicação entre módulos
  • São mais realistas que unitários
  • São mais rápidos que E2E
  • São essenciais para APIs e microserviços
Em sistemas distribuídos, integração é onde muitos bugs nascem.

Onde os testes entram?

Teste Unitário

Valida regras de negócio isoladas.

Teste de API

Valida contratos HTTP, payloads e respostas.

Teste E2E

Valida o fluxo completo do usuário.

Teste de Contrato

Garante compatibilidade entre serviços independentes.

Honeycomb Testing

Em microserviços, o maior valor normalmente está nos testes de integração e API.

  • Precisamos validar fronteiras
  • Precisamos validar contratos
  • Precisamos validar comunicação HTTP/gRPC
  • Precisamos validar comportamento distribuído

Problema Real do CI/CD

Imagine isso:

  • Banco de dados
  • RabbitMQ
  • Redis
  • API de pagamentos
  • API de autenticação
  • API de terceiros

Subir tudo isso em cada pipeline é lento, caro e instável.

Mocks vs Stubs

Stub

Retorna uma resposta fixa e previsível. 


axios.get.mockResolvedValue({
  data: {
    nome: 'Alessandro'
  }
});

Mock

Além de responder, também valida comportamento. 


expect(axios.get)
  .toHaveBeenCalledTimes(1);

Exemplo de API


const express = require('express');
const axios = require('axios');

const app = express();

app.get('/usuario/:id', async (req, res) => {

  try {

    const resposta = await axios.get(
      `https://api.github.com/users/${req.params.id}`
    );

    return res.json({
      nome: resposta.data.name,
      bio: resposta.data.bio
    });

  } catch (error) {

    return res.status(500).json({
      erro: 'Falha externa'
    });

  }

});

module.exports = app;

Testando sem acessar API real


const request = require('supertest');
const axios = require('axios');

const app = require('./app');

jest.mock('axios');

describe('GET /usuario/:id', () => {

  it('deve retornar usuário formatado', async () => {

    axios.get.mockResolvedValue({
      data: {
        name: 'Linus',
        bio: 'Linux creator'
      }
    });

    const response = await request(app)
      .get('/usuario/1');

    expect(response.statusCode)
      .toBe(200);

    expect(response.body)
      .toEqual({
        nome: 'Linus',
        bio: 'Linux creator'
      });

  });

});

O que estamos validando?

  • Status HTTP
  • Formato do payload
  • Mapeamento de dados
  • Tratamento de erro
  • Integração entre camadas

Tudo isso sem depender da internet.

Testando Cenários de Erro


it('deve retornar erro quando API externa falhar', async () => {

  axios.get.mockRejectedValue(
    new Error('timeout')
  );

  const response = await request(app)
    .get('/usuario/1');

  expect(response.statusCode)
    .toBe(500);

  expect(response.body)
    .toEqual({
      erro: 'Falha externa'
    });

});

Testes de Contrato

Consumer

  • Define expectativas
  • Payload esperado
  • Status esperado
  • Headers esperados

Provider

  • Valida compatibilidade
  • Garante não quebrar contratos
  • Protege deploys
Times conseguem evoluir APIs sem precisar subir tudo junto.

Pact e Consumer Driven Contract


npm install @pact-foundation/pact --save-dev
  • O consumidor gera um contrato
  • O provider valida esse contrato
  • Evita que mudanças silenciosas quebrem produção

Pipeline Ideal


name: ci

on:
  push:
    branches: [ main ]

jobs:

  tests:

    runs-on: ubuntu-latest

    steps:

      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4

      - run: npm install

      - run: npm test

Todo push executa testes automaticamente.

Boas Práticas

  • Não dependa de internet no CI
  • Teste cenários felizes e de erro
  • Evite mocks excessivos
  • Use dados previsíveis
  • Mantenha testes rápidos
  • Não misture E2E com teste de API
  • Teste contratos críticos
