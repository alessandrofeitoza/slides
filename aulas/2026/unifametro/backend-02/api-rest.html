O que é uma API?

Uma API é um contrato entre sistemas.

Ela define regras claras para comunicação.

Quanto melhor o contrato, menor o acoplamento.

APIs estão em todo lugar

  • Netflix
  • Nubank
  • Uber
  • iFood
  • WhatsApp
  • ERP corporativos

O problema das APIs mal projetadas


POST /createUser
POST /saveUser
POST /registerUser
POST /newUser
O problema não é funcionar.
O problema é manter.

O que significa REST?

  • Representational State Transfer
  • Separação cliente-servidor
  • Stateless
  • Cache
  • Recursos identificados por URI
  • Interface uniforme

REST não é CRUD

Um dos maiores equívocos do mercado.

CRUD é uma implementação.

REST é um conjunto de restrições arquiteturais.

Quando uma API se torna madura?

Nem toda API que utiliza HTTP é realmente REST.

Leonard Richardson propôs um modelo para medir a maturidade de APIs.

Richardson Maturity Model

0 → 3

Nível 0 — RPC

Nível 1 — Recursos

Nível 2 — HTTP Correto

Nível 3 — HATEOAS

Nível 0 — RPC


POST /createUser
POST /deleteUser
POST /updateUser
HTTP é apenas um meio de transporte.

Nível 1 — Recursos


/users
/orders
/products
Os recursos passam a ser identificados por URLs.

Nível 2 — Uso correto do HTTP


GET /users
POST /users
PATCH /users/1
DELETE /users/1
Aqui encontramos a maioria das APIs modernas.

Nível 3 — HATEOAS

A resposta da API informa quais ações podem ser executadas.

O cliente navega pela API através dos links retornados.

Exemplo de HATEOAS


{
  "id": 10,
  "nome": "João",
  "_links": {
    "self": "/users/10",
    "orders": "/users/10/orders",
    "delete": "/users/10"
  }
}

Vale a pena usar HATEOAS?

  • Pouco utilizado em APIs corporativas
  • Comum em APIs públicas complexas
  • Importante conhecer para entender REST completo

Idempotência

Uma operação idempotente pode ser executada várias vezes sem alterar o resultado final.

Exemplo


DELETE /users/10
DELETE /users/10
DELETE /users/10
O usuário continua removido.

Quais métodos são idempotentes?

Método Idempotente?
GET Sim
PUT Sim
DELETE Sim
POST Não

Por que isso importa?

  • Retries automáticos
  • Falhas de rede
  • Balanceadores de carga
  • Sistemas distribuídos

Versionamento de APIs

APIs evoluem.

Clientes antigos precisam continuar funcionando.

Estratégias de Versionamento

  • URL
  • Header
  • Content Negotiation

Versionamento por URL


/api/v1/users
/api/v2/users
Estratégia mais simples e mais utilizada.

Versionamento por Header


Accept: application/vnd.company.v2+json
Mantém URLs limpas, mas aumenta a complexidade.

Quando criar uma nova versão?

  • Mudança de contrato
  • Remoção de campos
  • Mudança de comportamento
  • Quebra de compatibilidade

SOLID

Cinco princípios para reduzir acoplamento.

Tornam sistemas mais fáceis de evoluir.

S — Single Responsibility Principle

Uma classe deve possuir apenas um motivo para mudar.

Violação do SRP


class UserService
{
    public function createUser(){}
    public function sendEmail(){}
    public function exportPdf(){}
    public function generateReport(){}
}

O — Open Closed Principle

Aberto para extensão.

Fechado para modificação.

L — Liskov Substitution Principle

Subtipos devem poder substituir seus tipos base.

I — Interface Segregation Principle

Interfaces pequenas são melhores que interfaces gigantes.

D — Dependency Inversion Principle

Dependa de abstrações.

Exemplo


class UserService
{
    public function __construct(
        UserRepositoryInterface $repository
    ) {}
}

Não faça isso


class UserService
{
    public function __construct(
        MySqlUserRepository $repository
    ) {}
}

Dependency Injection

Objetos recebem dependências.

Objetos não criam dependências.

Benefícios da DI

  • Menos acoplamento
  • Maior testabilidade
  • Facilidade para trocar implementações
  • Código mais limpo

JWT

JSON Web Token

Um dos mecanismos mais utilizados para autenticação em APIs.

Estrutura do JWT


HEADER
.
PAYLOAD
.
SIGNATURE

Exemplo de Payload


{
  "sub": 123,
  "name": "João",
  "role": "admin"
}

Importante

JWT não criptografa informações.

JWT apenas assina informações.

Nunca coloque dados sensíveis no payload.

Fluxo JWT

Problemas comuns com JWT

  • Expiração muito longa
  • Payload excessivo
  • Armazenamento inseguro
  • Ausência de refresh token
