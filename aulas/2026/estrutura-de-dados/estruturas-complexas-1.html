Além do obvio

Quando Arrays e Objetos não dão conta

1. O Paradoxo do Array

Arrays são ótimos para guardar coisas em ordem.

Mas e quando precisamos encontrar algo rápido?

A Busca Lenta


const cpfsCadastrados = ["111.111.111-11", "222.222.222-22", ...]; // 1 milhão de itens

// O JS vai olhar de gaveta em gaveta (Busca Linear)
const existe = cpfsCadastrados.includes("999.999.999-99");

Se o item for o último, ele testou 1 milhão de vezes!

A Solução: SET

O que é um Set?

  • É uma coleção de valores.
  • A busca nele é quase instantânea.
  • Estrutura mais "profissional"

E o pulo do gato?

Não aceita valores duplicados

Set na Prática


// Passamos o array para o Set
const cpfsSet = new Set(["111", "222", "111"]);

console.log(cpfsSet); // Set(2) { '111', '222' } (removeu a cópia)

// Busca ultra-rápida (O(1))
console.log(cpfsSet.has("222")); // true

2. O Paradoxo do Objeto Literal

Objetos `{}` são os dicionários do JS.


const notas = {
    "Joao": 8,
    "Maria": 10
};

Problema: As chaves de um objeto SÓ podem ser Strings ou Symbols.

E se eu quiser usar um Objeto como Chave de outro Objeto?

Entra em cena o MAP

O Map é um Dicionário turbinado.


const metadata = new Map();

const botaoLogin = document.getElementById('btnLogin');
const botaoSair = document.getElementById('btnSair');

// A CHAVE É O ELEMENTO HTML, O VALOR É UM OBJETO DE DADOS!
metadata.set(botaoLogin, { clicks: 0, lastClick: new Date() });

console.log(metadata.get(botaoLogin));

Por que a disciplina de Estrutura de Dados ama Maps e Sets?

Porque por debaixo dos panos, eles são Tabelas Hash.

Eles não percorrem (iteram) os dados para achar a resposta. Eles usam matemática para calcular o endereço exato na memória.

3. O Paradoxo das Strings Soltas

O famoso "Magic String"


let statusUsuario = "ATIVO";

// 3 meses depois, em outro arquivo...
function verificarPermissao(status) {
    // ERRO DE DIGITAÇÃO!
    if (status === "Ativo" || status === "ativo") { ... }
}

Tratar regra de negócio com string solta é pedir para ter bug em produção.

A Solução: Enums (Enumerações)

Criando um "cardápio" fixo de opções.


// Object.freeze trava o objeto. Ninguém pode alterar as chaves/valores.
const STATUS_USER = Object.freeze({
    ATIVO: 'ATIVO',
    INATIVO: 'INATIVO',
    BANNED: 'BANNED'
});

let statusAtual = STATUS_USER.ATIVO;

if (statusAtual === STATUS_USER.ATIVO) { ... }

4. Obsessão por Primitivos

A "Primitive Obsession"

Um CPF é uma String?

String aceita isso:


let cpf = "BATATA-DOCE"; // O JS não liga, é texto.

Value Objects (VO)

Dando "inteligência" ao dado bruto.


class CPF {
    constructor(numero) {
        if (numero.length !== 11) throw new Error("CPF Inválido");
        // Aqui teria a lógica real de validação
        this.valor = numero;
    }
}

// O sistema agora trabalha com a ESTRUTURA CPF, não com Strings soltas.
const clienteCpf = new CPF("12345678901");

Juntando as Peças (O Mundo Real)

Como isso tudo vira uma Estrutura de Dados de verdade?


// 1. O Value Object (O que guardar)
const cpf = new CPF("12345678901");

// 2. O Enum (O Estado)
const status = STATUS_USER.ATIVO;

// 3. A Estrutura Avançada (Map) para acesso ultra rápido
const bancoDeDados = new Map();

bancoDeDados.set(cpf.valor, { cpf, status });

Onde vamos usar isso no curso?

  • Nodes (Listas Encadeadas): São objetos que guardam Value Objects.
  • Grafos/Árvores: Usam Maps para conectar um nó ao outro.
  • Tabelas Hash: O princípio de funcionamento do Map e do Set.

O Código "Ruim" na Vida Real

Imagine um sistema de e-commerce processando pedidos.

A Armadilha das Strings Soltas


function enviarEmailCliente(pedido) {
    if (pedido.status === "Aguardando Pagamento") {
        console.log("Por favor, pague o boleto.");
    } else if (pedido.status === "pago") {
        console.log("Seu pedido está sendo separado!");
    } else if (pedido.status === "Cancelado") {
        console.log("Que pena, pedido cancelado.");
    }
}

// Em outra parte do sistema, um dev júnior cria o pedido:
const meuPedido = { id: 123, status: "Pago" };
enviarEmailCliente(meuPedido);

O que acontece com meuPedido?

NADA! 🐛

O Dev digitou "Pago" (com 'P' maiúsculo), mas o `if` esperava "pago" (minúsculo).

O e-mail não é enviado e o cliente fica sem saber de nada.

Por que não usar apenas Strings?

  • Erros de digitação: O JavaScript não avisa se você digitar "Canselado" com 's'.
  • Falta de Autocomplete: A sua IDE (VS Code) não sabe quais são os status possíveis. Você tem que adivinhar ou olhar a documentação.
  • Pesadelo no Refatoramento: Se o chefe pedir para mudar "pago" para "pagamento_aprovado", você terá que buscar e substituir em 50 arquivos diferentes.

A Solução: ENUM

Criando a nossa "Fonte da Verdade"


// Object.freeze impede que alguém modifique isso acidentalmente
const STATUS_PEDIDO = Object.freeze({
    AGUARDANDO: "AGUARDANDO_PAGAMENTO",
    PAGO: "PAGO",
    CANCELADO: "CANCELADO"
});

// Agora o VS Code vai dar AUTOCOMPLETE se você digitar STATUS_PEDIDO.

O Código "Bom" e Seguro


function enviarEmailCliente(pedido) {
    if (pedido.status === STATUS_PEDIDO.AGUARDANDO) {
        console.log("Por favor, pague o boleto.");
    } else if (pedido.status === STATUS_PEDIDO.PAGO) {
        console.log("Seu pedido está sendo separado!");
    } else if (pedido.status === STATUS_PEDIDO.CANCELADO) {
        console.log("Que pena, pedido cancelado.");
    }
}

// O dev não digita mais strings soltas!
const meuPedido = { id: 123, status: STATUS_PEDIDO.PAGO };
enviarEmailCliente(meuPedido);

O que ganhamos com o Enum?

  • Segurança: Se o Dev digitar STATUS_PEDIDO.PAGGO, o código quebra na hora avisando que isso não existe (Fail Fast).
  • Manutenção: Se precisar mudar o texto que vai pro banco, basta mudar em um único lugar (dentro do objeto STATUS_PEDIDO).
