Questão Teórica 1 — Importância dos Design Patterns

Durante o desenvolvimento de um sistema corporativo, uma equipe percebeu que diversos módulos apresentavam soluções semelhantes para problemas recorrentes de organização de código, acoplamento excessivo e dificuldade de manutenção. O líder técnico sugeriu a adoção de Design Patterns como estratégia para melhorar a qualidade estrutural do software.

Considerando esse contexto, assinale a alternativa correta:

  • A) São bibliotecas prontas que substituem a necessidade de implementação manual
  • B) São soluções reutilizáveis e documentadas para problemas recorrentes de projeto
  • C) São regras obrigatórias definidas por linguagens orientadas a objetos
  • D) São algoritmos focados exclusivamente em performance

Exemplo de Código — Composite (JavaScript)

class Component {
  render() {}
}

class Button extends Component {
  render() { console.log('Button'); }
}

class Panel extends Component {
  constructor() { super(); this.children = []; }
  add(child) { this.children.push(child); }
  render() { this.children.forEach(c => c.render()); }
}

Exemplo de Código — Composite (PHP)

interface Node {
    public function render(): void;
}

class File implements Node {
    public function render(): void { echo 'Arquivo'; }
}

Exemplo de Código — Composite Container (PHP)

class Folder implements Node {
    private array $children = [];

    public function add(Node $node): void {
        $this->children[] = $node;
    }

    public function render(): void {
        foreach ($this->children as $child) $child->render();
    }
}

Singleton — Conceito

O padrão Singleton garante que uma classe possua apenas uma única instância durante todo o ciclo de vida da aplicação, fornecendo um ponto global de acesso a essa instância.

  • Evita múltiplas instâncias desnecessárias
  • Centraliza acesso a recursos compartilhados
  • Muito usado com configuração, cache e conexão global

Singleton — Exemplos Práticos

Casos reais onde Singleton costuma aparecer:

  • Gerenciador de conexão com banco de dados
  • Serviço global de configuração da aplicação
  • Logger centralizado
  • Cache compartilhado em memória
  • Service Container / IoC em frameworks

Singleton — Exemplo em JavaScript

class AppConfig {
  constructor() {
    if (AppConfig.instance) {
      return AppConfig.instance;
    }

    this.theme = 'dark';
    AppConfig.instance = this;
  }
}

const a = new AppConfig();
const b = new AppConfig();

console.log(a === b); // true

Singleton — Exemplo em PHP

class DatabaseConnection
{
    private static ?self $instance = null;

    private function __construct() {}

    public static function getInstance(): self
    {
        return self::$instance ??= new self();
    }
}

Singleton — Cuidados no Uso

Apesar de útil, Singleton deve ser usado com cautela.

  • Pode introduzir estado global excessivo
  • Dificulta testes unitários
  • Pode esconder dependências da aplicação
  • Frequentemente é overused por iniciantes

Regra prática: use apenas quando realmente fizer sentido existir uma única instância.
